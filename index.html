What's new IN WILDFLY 9

Tomasz Adamski

WildFly

  • Formerly JBoss AS
  • Community project with goals of fraquent releases and rapid innovation
  • Upstream for JBoss EAP
  • Fast, lightweight, manageble runtime
  • Java EE7 compliant
  • Open source, community driven project
  • Version 9 coming soon!

HTTP/2

  • Reduction in page load time

Goals

  • No changes in network infrastructure
  • No changes in website content

HTTP/2

  • Binary protocol with request framing

Design

  • Multiplexed - many prioritized streams interleaved in one TCP connection
  • Header compression
  • Server push - server is able to push data to a client before it has asked for it
  • Stream flow control - prevents one stream from blocking the entire connection

HTTP/2

  • Although not required by specification major browsers require HTTP/2 to use TLS connections with ALPN extension

WildFly

  • Configuration:
    • configure server's security realm
    • install ALPN jar

Server graceful shutdown

  • Server can be suspended
  • After suspend operation is executed new requests are rejected but running requests are able to finish their work
  • Suspend operation ends when either all requests finish or timeout is extended
  • Usage example: graceful shutdown, configuration changes
  • Server can be resumed - request processing starts immediately

Server graceful shutdown

  • :suspend(timeout=x)

Management operations

  • :resume
  • :shutdown(timeout=x)
  • :read-attribute(name=suspend-state)

Server graceful shutdown

  • RUNNING - normal operation

States

  • PRE_SUSPEND - external parties are notitied that server is about to go away
  • SUSPENDING - all new requests are rejected
  • SUSPENDED - All existing requests have completed, server has suspended

Codebase split

  • The base architecture of WildFly has been fully separated, and is now versioned separately from the full platform https://github.com/wildfly/wildfly-core
  • wildfly full - subsystems implementing EE functionality https://github.com/wildfly/wildfly
  • We provide tooling to faciliate building WildFly extensions and composing them into a server

wildfly-core

  • Fully concurrent service container
  • Management layer, including domain mode
  • Modular class loading which enables multi-tenancy of applications
  • Pluggable hot deployment layer
  • For framework authors that want to build their own application runtime using the WildFly architecture

wildfly-web

  • new servlet-only distribution
  • adds Undertow http://undertow.io  and naming subsystems on top of wildfly-core
  • with all capabilities provided by core
  • useful for applications that prefer to roll-their-own frameworks

wildfly-full

  • Complete Java EE7 server distribution
  • Contains all subsystems available in WildFly
  • Build on top of wildfly-web

Feature pack

  • Jar describing server extension
  • Consist of modules and subsystems configurations and content files
  • Lightweight - no jars included
  • Can depend on other feature packs
  • Build by maven plugin based on configuration file
  • During build all module dependencies are checked

Server provisioning

  • Feature packs are used by provisioning library to create a server
  • The library can be used in standalone mode or be embedded in maven plugin
  • Merges modules, configuration and content files from all feature packs
  • Can provision thick or thin server

Example: RESTful distribution

  • Users can use above tools to provision server with only those parts of WildFly that are necessary in their use case
  • Example: lightweight server, based on wildfly-web,  with ability to deploy RESTful applications

WildFly Swarm

  • tool for building runnable fat-jars with embedded WildFly
  • created jars can contain only necessary parts of WildFly
  • build using maven plugin
  • microservices friendly

WildFly as loadbalancer

  • httpd based loadbalancer

mod_cluster

  • Dynamic configuration of httpd workers
  • Server-side load balance factor calculation
  • Fine grained web-app lifecycle control

WildFly as loadbalancer

  • WildFly can now be used as front end mod_cluster based load balancer
  • One tool can be used to manage front and backend servers
  • More efficient communication protocols, such as HTTP/2, can be used
  • OpenShift: public cloud PaaS responsible for provisioning, management and scaling of applications

Cloud

  • Architecture: gears, cartdridges, scalling, failover
  • WildFly 9 cartridge is provided - lets you create scalable cloud applications out of the box

Offline CLI support

  • Ability to embed a WildFly process inside the CLI process
  • Usage: management of the server using CLI commands, without having a running server

Switch to OpenJDK ORB

  • Downstream project of OpenJDK
  • Better interoperability with the JVM ORB and the Java EE RI
